▌ 1. 계정 / 입점 ✅ (골격)
브랜드·인플루언서가 플랫폼에 들어오는 단계. 활성이어야 로그인·연결·계약 가능(운영자 통제 지점 ①). 기존 입점신청·가입 목업을 상태머신으로 정리.

핵심 규칙
활성이어야 이용 가능 — 심사대기·반려 상태에선 연결·계약·판매 불가.
브랜드·인플루언서 공통 골격, 차이는 가입 시 받는 정보(위 표, [확장 가능]).
반려 사유 입력은 운영자 필수 — 신청자가 무엇을 고칠지 알 수 있도록 사유가 신청자에게 노출됨.
재신청은 같은 신청서 수정 방식 — 반려된 신청 건을 수정 가능 상태로 되돌려 보완·재제출(이력 누적). 새 신청서 작성 아님.
(구현 시) 입점_신청_상세 목업에 반려 사유 입력란 + 신청자 노출 영역 포함.
결정·메모
운영자 제재(계정 정지) 상태는 MVP 제외 (확정). sanction-manage 목업은 다음 단계.
재신청 횟수 제한 없음(베타). 정식 단계에서 "N회 반려 시 제한" 등 [확장 가능].
가입 정보 세부 항목 — [확장 가능]
▌ 2. 연결 (브랜드 ↔ 인플루언서) ✅
브랜드가 인플루언서를 지정해 만드는 계약 가능 관계. 계약·소통의 전제. 탐색·매칭 시스템 없음(외부 합의 전제).

핵심 규칙
연결됨이 계약·소통의 전제.
수락 단계 필수 — 인플루언서 모르게 연결되지 않도록 1-클릭 수락.
진행중 공구가 있으면 해제 제한 — 해당 공구 종료 후에만 해제 가능.
탐색·필터·정렬 없음 — 상대를 검색/코드로 지정만.
▌ 3. 소통 스레드 ✅
연결된 브랜드·인플루언서가 글을 주고받는 공간. 게시판형(실시간 아님, 안 읽은 수 표시).

핵심 규칙
'계약 시작'은 연결됨 상태에서만 — 아니면 버튼 비활성.
서명 순서: 인플루언서 먼저 → 브랜드 최종. 부분서명은 플래그로 추적.
발송 취소(회수) — 서명대기·인플루언서 서명 전이면 브랜드가 회수 → 작성중 복귀·수정·재발송(알림 #16). 서명 후엔 회수 불가(취소만).
체결완료가 공구 생성 게이트. 종료 상태(거절·만료·취소) 불가역 → 재계약은 신규 객체.
계약 1건 = 공구 1건. 철회 권한: 브랜드 + 운영자.
증거 보강은 A안 솔루션 — 문서ID·체결 PDF·감사추적인증서 참조만 보관.

핵심 규칙
계약 체결완료가 공구 생성 게이트 (계약 1:1 공구). 공구가·대상 상품 상속.
운영자 승인이 준비완료 게이트 (공구 오픈 승인, 운영자 통제 지점 ②).
공구 게시물 등록이 준비완료의 전제 — 공구 게시물 없으면 오픈(승인) 불가 → "필수 등록" 강제.
오픈 승인 시 문구 점검 결합(경량 검수) — 운영자가 준비완료 승인 때 상품 상세·공구 게시물 본문을 체크리스트로 점검(화장품법 금지 표현: 의약품 오인·기능성 표방·효능 단정 등 + 대가관계 표시 확인). 별도 상품 등록 검수 단계는 두지 않음 — 상품은 공구 게시물을 통해서만 소비자에게 노출되므로 이 게이트 하나로 노출 전 차단. 체크리스트 목록은 변호사 확정.
진행중 진입 = 준비완료 AND 기간 시작일 도래.
진행중에서만 소비자 구매 가능. 조기 마감 허용(운영자).
취소 → 환불 유발 — 공구 취소 시 해당 공구의 미확정 주문 전체를 운영자가 일괄 환불 처리(PG 취소), 이미 배송된 건은 회수 절차 준용. (환불 상세는 장바구니/주문·정산)

재고·품절은 옵션 단위. 상품 전체 품절 = 모든 옵션 0 (파생값).

두 축 독립. 진행중 상품 미진열은 운영자만, 평시 브랜드+운영자.
정가/공구가 분리 — 공구가는 계약 상속(수정 불가), 매출·정산은 공구가 기준.
▌ 7. 공구 게시물 ✅
계약 후 필수로 등록하는, 공구의 소비자 입구 게시물. 공구 상품을 보여주고, 누르면 공구가가 적용된 상품 상세페이지로 연결한다. 콘텐츠가 아니라 공구의 소비자 얼굴(공구와 1:1).

필수·1:1 — 공구당 반드시 1개, 준비완료 승인의 전제.
본문 글만, 상품 이미지·상세는 상품 상세페이지가 담당.
운영자만 숨김 — 공구의 핵심 자산이라 인플루언서 임의 숨김 불가.
예약·노출중 공구 게시물은 본문 수정 잠금 — 오픈 승인 시 점검한 문구의 무력화 방지. 수정 필요 시 운영자 경유. (일반 게시물은 자유 수정 유지)
공구 종료 = 자동 숨김(종료). 사람이 안 내려도 시스템이 처리.
상세페이지 = 게시물(리스팅) 단위 진입 — 가격은 정가 취소선 아래 "○○님의 공구가 + 공구 가격". 상품 직접 URL 진입 차단/리다이렉트.
묶음 진열 ≠ 묶음 강제 구매 — 한 게시물에 여러 상품이 묶여 있어도 소비자는 원하는 상품·옵션만 개별 구매 가능. 게시물은 진열 단위일 뿐, 구매 최소 단위는 상품(옵션).
경제적 대가관계 표시 자동 삽입 — 공구 게시물에 "인플루언서가 판매 수익을 배분받는 공동구매" 취지의 표시 배지/문구를 시스템이 자동 삽입(공정위 추천·보증 심사지침). 인플루언서가 제거·숨김 불가. 최종 문구는 변호사 검토.
판매자/추천인 역할 분리 표기 — 상세페이지에 "판매자: ○○브랜드 / 추천: ○○님" 구조로 표기(판매 주체 오인 방지, 분야 1 신원정보 표시와 연동).
장바구니 연동 — 항목이 이 게시물(쇼룸+공구)을 참조. 공구가는 공구 기간 내 불변 → 담을 때 가격 저장(스냅샷). 공구 종료 시 장바구니에서 자동 제거, 품절은 결제 시 확인. (상세는 장바구니/주문 객체)

핵심 규칙
결제 페이지에서 배송지·결제수단 확정 후 결제 — 장바구니 직접 결제 불가.
재고 차감 = 결제완료 시점 — 담을 때·결제 직전엔 차감하지 않음. 결제 시 재고 부족이면 결제 차단(초과판매 방지). 환불·취소 시 재고 원복.
준비 시작은 브랜드 수동 — 결제완료 → 상품 준비중은 브랜드가 "준비 시작"을 눌러야 전환. 이 전까지가 소비자 단순 취소 구간.
환불은 위험도에 따라 둘로 분리:
결제완료 단순 취소(물건 안 나감) → 소비자 취소 시 PG 자동 취소, 운영자 확인 불필요.
반품 환불(물건 나감) → 회수·입고 확인 필요, 환불 실행은 운영자(입고 확인 후 PG 취소).
반품 반송은 택배사 회수 연동(문 앞 수거) — 소비자는 회수 희망일·수거지만 입력, 회수 송장은 택배사가 자동 발급(소비자 송장 입력 없음).
환불 배송지 = 브랜드 반품 수취 주소(브랜드 사전 설정).
반송비 부담 주체 = 정책에서 정함 (단순변심 소비자 / 불량·오배송 브랜드 — 약관·청약철회 정책, 변호사 검토).
공구 취소·운영자 사유 환불은 단계 무관 가능 (소비자 보호 예외).
환불 완료 시 재고 원복.
결제 단위 ≠ 정산 단위 — 통합 결제해도 공구(게시물)별 주문 항목으로 분리 저장.
구매확정 = 배송완료 후 7일 자동(반품 없을 시) — 청약철회 기간과 정합. 구매확정이 정산 대상. *기산점: 택배사 배송완료 처리일을 수령일로 추정, 실제 수령일 소명 시 그에 따름(약관 명문화, 변호사 검토).*
반품 거절 가능 — 약관상 제한 사유(화장품 개봉·사용, 포장 훼손, 기간 경과 등) 해당 시 요청 단계(1차) 또는 입고 검수(2차)에서 거절. 거절 시 사유를 소비자에게 통지하고 상품 재발송(배송비 부담은 귀책 따라 — 정책).
개별 취소·환불 (단위 = 상품+옵션 SKU) — 다른 옵션은 별개 항목이라 각각 독립 취소·환불 가능(빨강만 환불, 파랑 유지). 같은 항목은 수량 부분 취소 가능(2개 중 1개). 환불액 = 공구가 × 취소 수량, 재고는 그만큼 원복. (반품 시 회수도 취소 수량 단위)
청약철회 기간 등 법정 기준은 약관·환불 정책에서 정함(변호사 검토).
[논의]
연동할 택배사/회수 API 사업자 선정 — 개발 착수 시 (확정 사항: 회수 연동 방식)

▌ 16. 알림 (발생 지점) ✅
별도의 무거운 객체가 아니라, 각 객체의 상태 전이 시 생성되는 인앱 알림. 베타 = 인앱 전용(푸시·이메일·알림톡 제외), 안 읽은 수 뱃지 + 알림 목록.
