▌ 0. 한 줄 정의
이번 베타 MVP는 "공동구매 거래 실행 루프" 하나만 검증한다.
브랜드·인플루언서는 외부에서 합의한 뒤 입점(운영자 승인)하고, 브랜드가 플랫폼에서 인플루언서를 직접 연결한다(탐색·매칭 시스템 없음). 브랜드는 상품을 정가로 상시 등록해 둔다. 시스템은 연결 → 계약(상품 선택·공구가 결정) → 공구 생성·공구가 적용 → 쇼룸 게시물(공구/일반) → 소비자 구매·CS → 정산까지의 실제 거래 흐름을 처리한다.
"공동구매 운영 OS"라는 전체 비전은 그대로 유지하되, 이번 베타는 그중 거래 루프 한 조각만 잘라 먼저 검증한다.

▌ 3. 확정된 핵심 결정
거래 루프가 "실제로 돌게" 만드는 핵심 결정. ②③ 문서의 직접 입력값이 된다.
[TABLE]
구분 | 결정 항목 | 확정 내용 | ②③에 미치는 영향
A | 인플루언서 리워드 기준 | 리워드율은 계약 체결 시 합의된 내용으로 확정된다 (브랜드가 제안 → 계약으로 확정) | 계약 객체에 리워드율 필드 · 정산은 이 값을 분배 기준으로 사용
D | 공구가(판매가) 기준 | 할인은 공구가 모델로 처리. 브랜드는 정가로 등록, 공구가는 계약에서 결정 → 해당 공구의 판매가로 적용(상품 정가와 분리). 쿠폰 미사용 | 상품은 정가 보유 · 공구가는 공구(리스팅) 단위 · 매출·정산은 공구가 기준
E | 상품 진입·공구가 귀속 단위 | 상품은 쇼룸 게시물(쇼룸+공구) 단위로만 진입하며 상세페이지도 그 단위(쇼룸마다 공구가 다를 수 있음, 정상). 공구가는 상품이 아니라 장바구니 항목이 참조하는 게시물에 묶임. 공구가는 공구 기간 내 불변 → 장바구니는 담을 때 가격 저장, 공구 종료 시 자동 제거·품절은 결제 시 확인 | 장바구니 항목에 게시물(쇼룸·공구) 참조 + 담을 때 공구가 저장 · 같은 상품도 쇼룸 다르면 별개 항목 · 상품 직접 URL 진입은 차단/리다이렉트
B | 결제·정산 처리 방식 | 소비자 결제는 PG 자동결제, 정산 이체(브랜드·인플루언서 송금)는 운영자 확인 후 실행 | 정산에 "정산 대기 → 운영자 확인 → 이체 완료" 상태 분기 필요
C | 계약 방식 | 외부 전자계약 솔루션 연동(A안) 으로 확정 (협의항목 입력 → 계약서 자동 생성 → 임베디드 서명) | 계약 객체에 문서ID·상태·체결 PDF·감사추적인증서 참조 저장 (증거 보강은 솔루션이 분담)
F | 연결·소통 방식 | 탐색·매칭 시스템 없음(외부 합의 전제). 브랜드가 파트너센터에서 인플루언서를 셀프 연결(검색·코드, 인플루언서 수락 필요). 연결당 게시판형 소통 스레드 1개(실시간 아님, 이미지·영상 첨부 가능(최대 20개), 안 읽은 수 표시) | 연결 객체·소통 스레드 객체 신규 · 운영자는 스레드 모니터링 열람
G | 운영자 통제 지점 | ① 입점(회원가입) 승인 ② 공구 오픈 승인(준비중→준비완료). 계약 단계는 운영자 개입 없음 | 입점 승인 흐름 + 공구 준비완료 게이트
[/TABLE]
3-1. 계약 방식 — 확정: A안 (외부 전자계약 솔루션 연동)
전자서명은 법적으로 유효하나(부록 A), 분쟁 시 "본인이 서명했다"는 입증 책임이 사업자에게 있어 증거력이 관건이다. 리워드(금전)가 계약에 직접 걸리는 구조라, 본인확인·무결성·감사추적을 검증된 솔루션에 위임하는 A안으로 확정한다.
[TABLE]
안 | 방식 | 증거력 | 상태
A | 외부 전자계약 솔루션 연동 (모두싸인·이싸인온 등) | 최강 (공인 감사추적인증서·제3자 보증) | ✅ 채택
B | 플랫폼 내 동의 기록 + 본인확인·타임스탬프 보강 | 중간 (입증 책임 자체 부담) | 미채택
C | 약식 동의 기록만 | 약함 | 미채택
[/TABLE]
A안 연동 흐름 (거래 루프 1단계 구현):
[TABLE]
단계 | 내용 | 비고
① 템플릿 등록 | 변호사 검토를 거친 마스터 계약서를 솔루션에 템플릿으로 등록, 협의항목을 요청자(문서편집) 입력 필드로 라벨링 | 협의항목 = 가변, 서명란 = 서명자 필드
② 생성·발송 | 계약 단계에서 협의항목 값을 주입해 계약서 자동 생성 → 브랜드·인플루언서를 서명자로 지정해 발송 | 템플릿 서명요청 API
③ 임베디드 서명 | 외부 이동 없이 파트너센터/쇼룸 스튜디오 안에서 서명 | 이탈 방지
④ 상태 동기화 | 서명 완료/거절 이벤트 수신 → 계약 상태 "대기 → 체결" 갱신 | Webhook + 상세조회 API
⑤ 증거 보관 | 체결 PDF·감사추적인증서를 우리 DB에 저장, 공구ID 기준 분류 | download URL + metadata
[/TABLE]

탈퇴 시 파기 vs 법정 보존 | 개인 식별정보 파기/비식별화 + 거래기록(주문·정산·계약)은 전자상거래법 기간 분리 보존. 항목별 기간은 변호사 확정
소통 스레드 열람·보존 고지 | 운영자 열람 사실을 입점 약관+화면에 상시 고지, 스레드(쌍 기준)의 해제 후 보존 목적·기간을 약관에 명시 — 변호사 패키지 포함
표시광고 (추천·보증 심사지침) | 공구 게시물에 대가관계 표시 자동 삽입(시스템, 제거 불가 — 문구는 변호사), 인플루언서 표시 의무 가이드라인을 입점 약관에 포함, 일반 게시물에서 공구 상품 언급 시 표시 의무 안내(온보딩). "○○님의 공구가" 오인 방지는 판매자/추천인 분리 표기로 해결
화장품법 표시·광고 (경량 검수 ⓑ) | 별도 상품 등록 검수 없음 유지. 대신 공구 오픈 승인 시 상품 상세·게시물 문구를 체크리스트로 점검(의약품 오인·기능성 표방·효능 단정 등 금지 표현 + 대가관계 표시). 상품이 공구 게시물로만 노출되므로 이 게이트 하나로 충분. + 입점 약관에 브랜드 표시·광고 책임 조항, 금지 표현 가이드(브랜드·인플루언서 통합)를 온보딩 자료로 — 체크리스트·조항 문구는 변호사
정산 후 환불(클로백) 대비 | 지급 유예 방식(ⓒ) — 정산 기한을 마지막 구매확정 + 영업일 5일로 설정(운영자 사유 환불이 드러나는 창 커버). 유예 후 극예외는 운영자 수동 협의, 약관에 차감·반환 근거 조항 포함(변호사 패키지). 유보금 제도는 [확장 가능]
소비자 계정·로그인 | 소셜/간편 로그인(운영자 승인 없음), 약관·개인정보 동의 필수
소비자 배송지(주소록) | 소비자 계정 하위, 여러 개 저장·기본 지정. 결제 페이지에서 선택/신규 입력
팔로우(소비자→인플루언서) | 단방향 팔로우. 팔로워 게시물 알림의 기반
인앱 알림 | 베타는 인앱 전용(푸시·이메일 제외). 거래 이벤트 + 팔로워 게시물 알림
재고 차감 규칙 | 결제완료 시 차감, 부족 시 결제 차단(초과판매 방지), 환불 시 원복
정산 자금 구조 (세무 + 전자금융) | A(우리가 직접 수취·분배) vs B(PG 지급대행·수수료만) — 세무(세금계산서·원천징수)만이 아니라 전자금융거래법 이슈: 직접 수취·보관·분배는 PG업/결제대금예치업 등록 대상일 수 있음. B(PG 하위몰·지급대행) 구조가 양쪽에서 안전 → PG사에 "브랜드+인플루언서 분배 가능 여부·지급 주기" 확인, 변호사에 적법성 확인. 정산 기한(영업일 3일)은 PG 지급 주기 확인 후 확정. 세무사 질문 리스트: ① A/B 구조별 우리 매출 인식 범위 ② 수수료 2% 세금계산서 발행 상대·시점 ③ 인플루언서 리워드 지급 처리(개인 원천징수 / 개인사업자 / 법인별) ④ 브랜드 매출 세금계산서·현금영수증 의무의 주체 ⑤ 카드·간편결제 한정 시 현금영수증 의무 존부 ⑥ 부가세·원천세 신고 주기·운영 체크리스트
결제수단 범위 | 베타 결제수단 결정 필요 — 카드·간편결제 한정 권장(계좌이체·가상계좌 등 현금성 결제는 결제대금예치(에스크로) 의무 발생 → 지원 시 PG 에스크로 서비스 적용)
[/TABLE]
▌ 5. 성공 기준 (베타가 "됐다"의 최소 조건)
9월 첫 공구가 계약부터 정산까지 끊김 없이 도는 것. 구체적으로:
5+5가 입점 승인 후 브랜드–인플루언서 셀프 연결로 매칭되고, 계약이 전자서명으로 체결된다.
공구 1건 이상이 계약부터 정산까지 무중단 완주한다.
소비자 결제·공구가 적용이 정상 동작하고, 상품 문의(CS)가 플랫폼 안에서 응대된다.
정산 금액이 §3-A(계약 리워드율) 기준으로 분배되고, §3-B 절차(운영자 확인 후 이체)로 집행된다.

▌ 6. 범위 경계 원칙 (왜 넣고 왜 빼는가)
거래가 실제로 도는 최소 루프만 검증한다. 판단 기준은 "이게 없으면 9월 첫 공구가 계약부터 정산까지 한 바퀴 못 도는가?"이고, 답이 '아니오'면 이번 베타에서 뺀다. 계약·CS는 이 기준상 루프 완주에 필요하므로 In, 탐색·매칭·리포트·추천은 운영자 수동 또는 다음 단계로 미룬다.
▌ 7. 다음 단계 연결
이 범위가 확정되면 → ② 거래 플로우 & 상태 정의서 → ③ 도메인 모델·비즈니스 규칙 순으로 내려간다.
②③은 확정본이 아니라 김화창에게 넘길 논의용 초안으로 만들고, 합의가 필요한 지점은 [논의] 태그로 표시한다. 물리 ERD·스키마·API는 김화창이 ②③을 받아 확정한다.

증거력이 관건 — 공인인증서 폐지로 "서명 진정성 추정" 조항이 삭제됨. 분쟁 시 "본인이 서명했다"는 입증 책임이 사업자에게 있음. 단순 이름 입력·클릭 동의는 부인 위험.
보강 요건 — ① 본인확인(휴대폰·카카오 등 인증) ② 타임스탬프·해시(무결성) ③ 감사추적(누가·언제·IP·기기·인증수단) ④ 보존(전자문서법 제4조: 열람 가능·원본 형태 보존).
권고 — 자체 인증 인프라 구축 대신 검증된 전자계약 솔루션 연동이 증거력·속도 면에서 유리.
